TECHNICAL REFERENCE / EGO CLI V0.1.0

Arquitectura de Ego CLI

Ego es una herramienta local de apoyo a decisiones técnicas. Ejecuta varios CLI de IA instalados y autenticados en el equipo, coordina una revisión estructurada entre ellos y guarda tanto el resultado como la resolución humana. Esta página documenta los componentes, límites y flujos implementados en la versión actual.

01 / SYSTEM MAP

Componentes y límites del sistema

Las interfaces interactivas y automatizadas comparten el mismo harness. El harness contiene el protocolo de deliberación, los contratos de los participantes, la ejecución protegida y el registro auditable. Los CLI externos y el workspace quedan fuera de ese límite y solo se accede a ellos mediante adaptadores controlados.

02 / RUNTIME

Flujo de ejecución de una consulta

La TUI y los comandos no interactúan directamente con los modelos. Ambos crean una solicitud para el harness, que persiste el run antes de comprobar participantes y comenzar el protocolo.

  1. 00
    Entrada

    La interfaz interpreta texto normal como consulta y reserva el prefijo / para comandos.

  2. 01
    Creación del run

    Se normaliza la ruta del workspace, se registra el estado Git inicial y se crea el run en SQLite.

  3. 02
    Comprobaciones previas

    Se verifican binario, versión, opciones requeridas, autenticación y protección Seatbelt.

  4. 03
    Deliberación

    El engine ejecuta los participantes en paralelo y espera a que cada fase termine antes de iniciar la siguiente.

  5. 04
    Finalización

    Se revalidan las citas, se compara el estado Git final y se crea el registro de decisión.

  6. 05
    Resolución humana

    La persona puede aceptar, elegir una alternativa, registrar su propia conclusión, diferir o rechazar.

03 / DELIBERATION PROTOCOL

Protocolo de deliberación

El engine organiza cinco barreras de fase. Cada fase define el contexto disponible, el schema de salida y si el participante puede leer archivos. Todos los participantes mantienen la misma autoridad y la pareja sintetizadora rota en cada run.

POSITION

Independent reasoning

Cada participante analiza la misma pregunta y puede inspeccionar el workspace sin recibir posiciones ajenas.

Qué se valida en cada respuesta

Cada respuesta debe cumplir el schema correspondiente. Las revisiones requieren una explicación suficiente; las síntesis deben referenciar argumentos existentes; y la reconciliación debe declarar explícitamente equivalencia o conflicto.

Una respuesta inválida recibe un intento correctivo con el error de validación. Un segundo fallo cierra el turno como fallido y el resultado se degrada según las respuestas que sigan disponibles.

Qué razonamiento puede consultar la persona

La timeline permite consultar posiciones independientes, revisiones, síntesis y reconciliaciones normalizadas. No muestra razonamiento privado ni la salida cruda del proveedor. Las revisiones individuales entre modelos se conservan en el registro, pero no se incluyen en la vista predeterminada.

04 / DECISION LOGIC

Estados y reglas de finalización

El estado final depende del número de respuestas válidas, del resultado de la reconciliación y de la vigencia de la evidencia. El acuerdo entre modelos no eleva por sí solo la confianza por encima de moderate.

AFTER PROBES ¿Cuántos participantes completan una posición?
0

Failed

Ningún participante completó una posición válida; el run termina sin recomendación.

1

Completed · low

La posición se conserva con confianza baja y una advertencia de consulta individual.

2+

Continúa

El run continúa con revisión cruzada, actualización de posiciones, síntesis y reconciliación.

RECONCILIATION ¿Dos reconciliaciones válidas reconocen equivalencia material?
YES

Completed · moderate max

Se usa como base la síntesis con mayor cantidad de evidencia válida. El resultado se guarda como completado y la confianza máxima permanece en moderate.

NO / UNKNOWN

Contested · low

Las propuestas se guardan como alternativas junto con sus conflictos. El registro queda pendiente de una resolución humana y no se aplica una votación automática.

FAILURE PATH ¿La síntesis queda incompleta o la evidencia crítica cambia?
INCONCLUSIVE

El resultado pasa a inconclusive cuando no se completan dos síntesis o cuando la revalidación detecta evidencia crítica obsoleta o inválida. También se aplica si un resultado con citas deja de conservar al menos una cita válida.

HUMAN RESOLUTION

Resolución humana de resultados

La resolución se guarda como un registro adicional y no modifica el resultado generado por los modelos. Para un resultado contested, aceptar requiere seleccionar una alternativa o registrar una conclusión propia; deferir y rechazar siguen disponibles.

/choose NAceptar alternativa
/decide TEXTConclusión propia
/deferEsperar evidencia
/rejectRechazar resultado

05 / SAFETY BOUNDARY

Modelo de ejecución de solo lectura

Los participantes inspeccionan el workspace en su ubicación real. Ego exige una prueba funcional de Seatbelt antes de ejecutar cualquier modelo y rechaza el participante si no puede confirmar que las lecturas funcionan y las escrituras están bloqueadas.

⌂

TARGET WORKSPACE

Real directory

Lecturas y búsquedas permitidas; modificaciones denegadas.

file-write* denied
WALL 01

Provider-native controls

Cada adaptador activa los modos de planificación o solo lectura disponibles en su CLI.

WALL 02

External macOS Seatbelt

Todas las llamadas se envuelven con un perfil que niega escrituras en el workspace.

WALL 03

Process constraints

El proceso recibe variables allowlist, timeout, límite de salida y captura separada de stdout/stderr.

CODEX
ONLY

DOCUMENTED EXTERNAL-SANDBOX VARIANT

Compatibilidad de Codex con el sandbox externo

El sandbox interno de Codex no puede anidarse dentro del Seatbelt aplicado por Ego. Su adaptador declara requires_external_sandbox y activa el modo externo del CLI. El runner comprueba que la llamada siga envuelta por Seatbelt y añade protección de escritura para las ubicaciones durables del usuario y del sistema.

  • CODEX_HOME temporal y aislado por llamada.
  • Copia privada 0600 de auth.json; se elimina al finalizar.
  • Configuración, sesiones, memoria, apps, browser y multi-agent desactivados.
  • Solo recibe sus propias credenciales; nunca variables de otros proveedores.
OPEN
CODE

ISOLATED EXTERNAL-SANDBOX ADAPTER

OpenCode conserva su modelo predeterminado

OpenCode no ofrece un sandbox de seguridad nativo. Ego lo ejecuta desde un proyecto temporal neutro, con HOME y XDG aislados, configuración sanitizada y Seatbelt externo. El adaptador no fija un modelo: conserva la selección predeterminada de OpenCode.

  • Credenciales y estado de selección copiados con permisos privados.
  • Plugins, MCP, agentes, comandos, shell, edición, web y delegación desactivados.
  • Lectura, glob y grep del workspace solo durante las fases con evidencia.
  • Tokens reportados; el coste calculado no se presenta como facturación confirmada.

06 / EVIDENCE CONTRACT

Alcance de la validación de evidencia

La validación confirma que la ruta, el rango de líneas y los hashes corresponden al workspace inspeccionado. No verifica automáticamente que la explicación del modelo sea una interpretación correcta del código.

01

Path boundary

La ruta se normaliza y debe quedar dentro del workspace seleccionado.

02

Exact lines

El rango indicado debe existir y se extrae exactamente para calcular su hash.

03

Two hashes

El registro incluye SHA-256 del archivo completo y del fragmento citado.

04

Final revalidation

Las citas y el estado Git se comparan nuevamente antes de crear la decisión.

EGO CAN VERIFY

  • Ruta
  • Rango de líneas
  • Hash del archivo
  • Hash del fragmento
≠

EGO DOES NOT PROVE

  • Interpretación semántica
  • Ausencia total de riesgos
  • Verdad por consenso
  • Corrección futura
